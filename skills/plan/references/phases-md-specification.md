# phases.md Specification

Complete schema defining every section of the generated `phases.md` file. The feat skill uses this specification to ensure consistent, complete output.

---

## Document Structure

```
phases.md
├── Header Block (+ optional Version wrapper for large projects)
├── Project Strategy (optional — testing archetype, AI features)
├── Design Decisions Log
├── Environment & Tech Stack Audit
├── Phase 0: Prerequisites & Setup
├── Phase 1 through Phase N
├── Critical Files
├── Consistency Rules
├── Key Learnings Protocol
└── Version Control Protocol
```

---

## Section 1: Header Block

The header provides metadata about the generated file.

**Required fields:**

| Field | Format | Example |
|-------|--------|---------|
| Project Name | Plain text | "TaskFlow — Project Management App" |
| Generated | ISO 8601 date | 2025-01-15 |
| Source File | Relative path | `./feature.md` |
| Total Phases | Integer (including Phase 0) | 7 |
| Generated By | Skill identifier | feat skill v1 |

**Format:**

```markdown
# phases.md — {Project Name}

| Field | Value |
|-------|-------|
| Generated | {YYYY-MM-DD} |
| Source | `{path-to-source-file}` |
| Phases | {N} (Phase 0 + {N-1} feature phases) |
| Generated By | feat skill |
```

**Optional fields (added by Phase 1c when applicable):**

| Field | Format | When Present |
|-------|--------|--------------|
| Version | Integer | When version decomposition is used |
| Version Scope | Plain text | When version decomposition is used |

**Version wrapper format (for large projects):**

When a project uses version decomposition (>10 phases or >30 requirements), phases.md includes a Version wrapper. Each version is a self-contained phases.md section:

```markdown
# phases.md — {Project Name}

| Field | Value |
|-------|-------|
| Generated | {YYYY-MM-DD} |
| Source | `{path-to-source-file}` |
| Versions | {V} |
| Generated By | plan skill |

## Version Roadmap

| Version | Scope | Phases |
|---------|-------|--------|
| V1 | {Simplest viable product} | Phase 0-{N} |
| V2 | {Next capability layer} | Phase 0-{M} |

---

# Version 1 — {V1 Scope Name}

| Field | Value |
|-------|-------|
| Phases | {N} (Phase 0 + {N-1} feature phases) |

{... standard phases.md sections follow for this version ...}
```

Each version gets its own Phase 0 (may reference prior version's setup), Design Decisions Log, phases, Critical Files, and Consistency Rules. Versions are independently shippable.

## Section 1b: Project Strategy (Optional)

Added during Phase 1c when the project uses a non-standard testing archetype or includes AI features. Consumed by /dev to adjust task execution.

**Format:**

```markdown
## Project Strategy

| Field | Value |
|-------|-------|
| Testing Archetype | A (Standard) / B (Hardware) / C (Perception) / D (Data Pipeline) / E (Compiled) / F (Mobile) |

| Stack Pack | stacks/{name}.md, or "None" — loaded from `_shared/references/stacks/` |
| AI Features | {count, or "None"} |
```

**AI Feature Inventory (when AI features exist):**

```markdown
### AI Feature Inventory

| Feature | Determinism Tier | Guard Mechanism | Notes |
|---------|-----------------|-----------------|-------|
| {feature name} | Tier 1 (Deterministic) | Structured output + schema validation | {notes} |
| {feature name} | Tier 2 (Reasoning) | RAG + citation enforcement | {notes} |
| {feature name} | Tier 3 (Creative) | Content filter + quality scoring | {notes} |
```

## Section 2: Design Decisions Log

Records every decision made during the feat workflow that was NOT in the original design document. This provides traceability for why certain choices were made.

**Required columns:**

| Column | Description |
|--------|-------------|
| Decision | What was decided |
| Options Considered | What alternatives existed |
| Chosen | Which option was selected |
| Rationale | Why this option was chosen |
| Decided By | "User" or "Default" |

**Format:**

```markdown
## Design Decisions Log

| # | Decision | Options Considered | Chosen | Rationale | Decided By |
|---|----------|--------------------|--------|-----------|------------|
| 1 | Testing framework | Jest, Vitest, Mocha | Vitest | Fast, ESM-native, Vite-compatible | User |
| 2 | Package manager | npm, yarn, pnpm | pnpm | Already installed, faster installs | User |
```

## Section 3: Environment & Tech Stack Audit

Complete results from the environment scanning phase.

**Required columns:**

| Column | Description |
|--------|-------------|
| Category | Runtime, Package Mgr, Framework, Deploy, Database, VCS, OS |
| Tool | Tool name |
| Required | Version constraint from design (or "any") |
| Found | Detected version (or "NOT FOUND") |
| Path | Filesystem path to binary (or "-") |
| Status | OK, MISSING, WRONG_VERSION |

**Format:**

```markdown
## Environment & Tech Stack Audit

| Category | Tool | Required | Found | Path | Status |
|----------|------|----------|-------|------|--------|
| OS | macOS | - | 14.2 | - | OK |
| OS | Architecture | - | arm64 | - | OK |
| Runtime | Node.js | >=18 | v20.11.0 | /usr/local/bin/node | OK |
| Package Mgr | pnpm | >=8 | 8.14.0 | /usr/local/bin/pnpm | OK |
| VCS | Git | any | 2.43.0 | /usr/bin/git | OK |
| Deploy | Docker | any | NOT FOUND | - | MISSING |
```

**Status rules:**
- `OK`: Tool found, version meets requirement
- `MISSING`: Tool not found on system
- `WRONG_VERSION`: Tool found but version doesn't meet requirement

## Section 4: Phase 0 — Prerequisites & Setup

Phase 0 is always present and always first. It contains ONLY scaffolding, installation, and configuration — zero feature code.

**Required subsections:**

### Goal
1-2 sentences describing what Phase 0 sets up. Always imperative form.

Example: "Initialize the project repository, install all dependencies, configure development tooling, and verify the development environment is fully functional."

### Scope

```markdown
### Scope

**In-scope:**
- Project initialization (scaffolding command)
- Dependency installation
- Configuration files (tsconfig, eslint, prettier, etc.)
- Development server setup
- Git initialization and .gitignore
- Directory structure creation

**Out-of-scope:**
- Any feature code
- Any business logic
- Any UI components beyond scaffolding defaults
```

### Tasks

Checkbox list. Each task follows this format:

```markdown
### Tasks

- [ ] **Task 0.1: Initialize project**
  - Run `npx create-next-app@latest taskflow --typescript --tailwind --eslint --app --src-dir`
  - **AC**: Given the command runs, When it completes, Then `taskflow/` directory exists with `package.json` containing `next`, `react`, `typescript` dependencies
  - **Files**: `package.json`, `tsconfig.json`, `next.config.js`, `src/app/layout.tsx`, `src/app/page.tsx`

- [ ] **Task 0.2: Install additional dependencies**
  - Run `pnpm add {packages}` and `pnpm add -D {dev-packages}`
  - **AC**: Given dependencies are installed, When `pnpm ls` is run, Then all packages appear in the list with correct versions
  - **Files**: `package.json`, `pnpm-lock.yaml`
```

### Validation Gates

```markdown
### Validation Gates

| Check | Command | Expected Result |
|-------|---------|-----------------|
| Dependencies installed | `pnpm ls` | All packages listed, no missing |
| TypeScript compiles | `npx tsc --noEmit` | Exit code 0, no errors |
| Dev server starts | `pnpm dev` | Server running on localhost:3000 |
| Lint passes | `pnpm lint` | Exit code 0, no errors |
| Build succeeds | `pnpm build` | Exit code 0, build output generated |
```

### Key Learnings Checkpoint

```markdown
### Key Learnings Checkpoint

After completing Phase 0, create `key-learnings/key-learnings-00.md` documenting:
- Exact versions of all installed tools and dependencies
- Any configuration decisions made (e.g., chose App Router over Pages Router)
- Directory structure established
- Development workflow commands (dev, build, test, lint)
- Any issues encountered during setup and how they were resolved
```

## Section 5: Phase N (Feature Phases)

Each feature phase (Phase 1 through Phase N) follows this structure.

### Goal
1-2 sentences. Imperative form. Describes user-facing value.

Example: "Implement user authentication with email/password login, registration, and session management."

### Context
References to prior phase outputs and key-learnings files.

```markdown
### Context

> **Required reading before starting this phase:**
> - `key-learnings/key-learnings-{N-1}.md`
> - Phase {N-1} validation gates must all pass

This phase builds on:
- [Specific output from Phase N-1 that this phase uses]
- [Configuration/pattern established in earlier phase]
```

### Scope

```markdown
### Scope

**In-scope:**
- [Specific capability 1]
- [Specific capability 2]
- [Specific capability 3]

**Out-of-scope (deferred to Phase {N+1} or later):**
- [Capability explicitly excluded]
- [Capability explicitly excluded]
```

### Tasks

Same format as Phase 0 tasks, but with feature-level granularity:

```markdown
### Tasks

- [ ] **Task {N}.1: {Task title}**
  - {Implementation instruction — imperative, specific, actionable}
  - **AC**:
    - Given {precondition}, When {action}, Then {expected result}
    - Given {precondition}, When {action}, Then {expected result}
  - **Files**: `{path/to/file1}`, `{path/to/file2}`
  - **Notes**: {Relevant patterns, libraries, gotchas, references to key-learnings}
```

**Task rules:**
- 3-8 tasks per phase
- Each task has at least one acceptance criterion
- Each task lists all files it creates or modifies
- Implementation notes reference specific libraries, patterns, or conventions
- Tasks are ordered by dependency (task N.2 may depend on N.1)

### Validation Gates

```markdown
### Validation Gates

| Check | Command | Expected Result |
|-------|---------|-----------------|
| Tests pass | `pnpm test -- --filter=auth` | All auth tests pass |
| Types check | `npx tsc --noEmit` | No type errors |
| Build succeeds | `pnpm build` | Clean build |
| Manual: Login flow | Open browser → /login | Form renders, submit works |
| Manual: Registration | Open browser → /register | Can create account |
```

**Gate types:**
- **Automated**: Shell commands with expected exit codes/output
- **Manual-visual**: "Open [URL], verify [visual element] appears"
- **Manual-behavioral**: "Perform [action], verify [behavior] occurs"
- **Integration**: "Complete [user flow] end-to-end"

### Key Learnings Checkpoint

```markdown
### Key Learnings Checkpoint

After completing this phase, create `key-learnings/key-learnings-{NN}.md` documenting:
- Architecture decisions made and rationale
- Patterns established (with file references)
- Issues encountered and resolutions
- Dependencies and versions locked
- Conventions established
- Notes for the next phase
```

## Section 6: Critical Files

A complete index of all files created or modified across all phases.

**Required columns:**

| Column | Description |
|--------|-------------|
| File Path | Relative path from project root |
| Created In | Phase number where file is first created |
| Modified In | Phase numbers where file is later modified |
| Role/Purpose | What this file does |

**Format:**

```markdown
## Critical Files

| File Path | Created In | Modified In | Role/Purpose |
|-----------|-----------|-------------|--------------|
| `src/app/layout.tsx` | Phase 0 | Phase 1, 3 | Root layout with providers |
| `src/lib/auth.ts` | Phase 1 | Phase 3 | Authentication utilities |
| `src/components/LoginForm.tsx` | Phase 1 | - | Login form component |
```

## Section 7: Consistency Rules

Project-wide conventions that every phase must follow. These rules compound — later phases inherit all rules from earlier phases.

**Format:**

```markdown
## Consistency Rules

These conventions apply to ALL phases. dev agents must follow these rules in every phase.

### Naming Conventions
- Components: PascalCase (e.g., `LoginForm.tsx`)
- Utilities: camelCase (e.g., `formatDate.ts`)
- Constants: SCREAMING_SNAKE_CASE (e.g., `MAX_RETRIES`)
- CSS: kebab-case or Tailwind utility classes

### File Organization
- Components: `src/components/{feature}/`
- Pages/Routes: `src/app/{route}/`
- Utilities: `src/lib/`
- Types: `src/types/`
- Tests: co-located with source (e.g., `Component.test.tsx`)

### Patterns
- [Pattern 1]: [Description and when to use]
- [Pattern 2]: [Description and when to use]

### Error Handling
- [Strategy for error handling across the project]

### Testing
- [Testing strategy and conventions]
```

## Section 8: Key Learnings Protocol

Standard text explaining the key-learnings workflow for dev and qa agents.

```markdown
## Key Learnings Protocol

### For dev Agents
1. Before starting Phase N, read ALL key-learnings files from Phase 0 through Phase N-1
2. After completing Phase N, create `key-learnings/key-learnings-{NN}.md`
3. If your implementation diverges from phases.md (e.g., a different approach was needed), document the divergence in key-learnings
4. Key-learnings take precedence over phases.md when they conflict — they reflect actual implementation reality

### For qa Agents
1. Read the phase's key-learnings file before validating
2. After validation, update the key-learnings file with any issues found and resolutions applied
3. Add a "qa Notes" section to the key-learnings file

### Key Learnings File Location
`{project-root}/key-learnings/key-learnings-{NN}.md`

Where `{NN}` is the zero-padded phase number (00, 01, 02, ...).

### Conflict Resolution
If a key-learnings file contradicts phases.md:
- Key-learnings wins (it reflects what was actually built)
- The next phase must follow the key-learnings version
- Document the divergence clearly for traceability
```

## Section 9: Version Control Protocol

Standard text explaining the commit and README workflow for dev and qa agents.

```markdown
## Version Control Protocol

### Commit Timing
- /dev commits all code changes after Phase Completion (Stage 7) — after verification passes, before the summary
- /qa commits any code fixes applied during validation — only if project files were modified
- Commits happen automatically at phase completion. Do NOT push to remote unless the user explicitly requests it.

### Commit Message Style
- Imperative mood, lowercase (e.g., "add sensor pipeline and perception layer")
- No structured prefixes (Fix:, Feat:) unless the project uses conventional commits
- No AI attribution — no Co-Authored-By, no "generated by", no AI-related text
- Keep concise: one line describing the phase deliverable

### README.md
- Updated by /dev after each phase completion to reflect current project capabilities
- Updated by /qa only if post-qa fixes changed user-facing behavior
- Contents: project purpose, current capabilities, requirements, setup, how to run
- No AI attribution or generation notices
```
