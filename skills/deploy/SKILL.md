# deploy — Deployment Readiness & Release Engine

> **EXECUTABLE WORKFLOW** — not reference material. Execute stages in order.
> Do not skip stages. Do not proceed without user confirmation at gates.

Verify pipeline completion, generate release artifacts (checklist, changelog), execute deployment, and run post-deploy verification. The "last mile" skill that bridges tested code to shipped product.

---

## Workflow

**Entry**: Follow the Skill Entry Protocol:
1. Read pipeline-state.md (verify clean state)
2. Read `brain.md` (self-model: strengths to lean into, gaps to watch for)
3. Verify prerequisites

### Stage 1: Pipeline Verification

**Blocking Conditions** — stop immediately if any are true:
- `phases.md` does not exist at project root
- Any phase has unchecked tasks
- Any qa report verdict is FAIL (not CONDITIONAL PASS — FAIL)
- User has not acknowledged any pipeline-state resume prompt

1. Parse `$ARGUMENTS` for deployment target or version. If not provided, detect from project config (vercel.json, package.json "scripts", Dockerfile, etc.). Each `/deploy` invocation handles one deployment target — for multi-target deployments (e.g., frontend + backend), invoke separately.

1a. **Complexity routing**: Count phases in `phases.md`, detect deployment target type, and assess deployment scope:
    - **Express** (1-2 phases AND target is static site or simple npm package): Stage 2 runs build+test+typecheck+security audit only (skip code quality deep scan and TODO/FIXME enumeration). Stage 3 generates compact changelog (Added + Fixed sections only). Stage 5 runs homepage load + console error check only.
    - **Full** (3+ phases, OR serverless/container target, OR multi-target, OR user passes `--thorough`): All stages run completely. Serverless targets (Vercel functions, AWS Lambda) always route Full — CORS, env-var, and cold-start issues require deep Stage 5 verification. If `product-readiness-report.md` exists at project root (generated by /polish), its findings are integrated into the release checklist at Stage 2.
    Within Full mode, adapt check depth by detected risk signals:
      - If key-learnings mention auth, payment, PII, or security → expand Stage 2 security checks (invoke `security-auditor` subagent, not just `npm audit`)
      - If any qa report was CONDITIONAL PASS → expand Stage 2 checks in the flagged category (e.g., CONDITIONAL PASS on accessibility → add accessibility-specific checks)
      - Stage 5 smoke test count scales with phase count: 2-3 phases → 2 critical journeys; 4+ phases → top 3-5 journeys from qa Category C
    Present detected mode to user before proceeding.

1b. Read `pipeline-state.md` at project root. If it exists and "Last Skill" is "deploy":
    - Present resume summary via `AskUserQuestion`: "Pipeline state shows deploy was last active. Resume deployment or restart?"
    - If user chooses Resume: skip to the last incomplete stage
    - If user chooses Restart: proceed normally
    If `pipeline-state.md` does not exist or "Last Skill" is not "deploy", proceed normally.
    See `~/.claude/skills/_shared/references/pipeline-state-protocol.md` for full resume logic.

2. Read `phases.md` at project root. If not found, STOP: "No `phases.md` found. Is there a pipeline to deploy? If this is a non-pipeline project, describe what you want to deploy."

3. Verify pipeline completeness — chain verification: each link (phases → key-learnings → qa reports) must hold before the next is checked.
   - Read ALL phase task checkboxes. Every task must be `- [x]`.
   - If any tasks are unchecked: STOP with list of incomplete tasks.

4. Read ALL key-learnings files (`key-learnings/key-learnings-00.md` through the last phase).
   - Verify each exists and is non-empty.
   - Extract all "Notes for Next Phase" — check if any have unresolved items.

5. Read ALL qa reports (`qa-reports/qa-report-phase-*.md`).
   - Verify each exists for every phase that has key-learnings.
   - Check verdicts: all must be PASS or CONDITIONAL PASS.
   - If any FAIL: STOP with: "Phase {N} qa report shows FAIL. Resolve blocking issues before deploying."
   - If CONDITIONAL PASS: compile the non-blocking issues for the release checklist.

6. Read `fix-log.md` if it exists (also check `hotfix-log.md` for legacy data). Compile all fixes applied since the last deployment.

6b. If `pipeline-state.md` exists, cross-reference its Phase Progress table against the phases.md/key-learnings/qa report data for consistency. Flag any mismatches to the user (e.g., pipeline-state shows a phase as COMPLETE but key-learnings is missing, or qa status doesn't match the qa report verdict).

7. Present pipeline summary:
   ```
   deploy Pipeline Verification

   Phases: {total} ({all complete / X incomplete})
   qa Reports: {count} ({all PASS / X CONDITIONAL PASS})
   Hotfixes since last deploy: {count}
   Outstanding items from CONDITIONAL PASS reports:
   - {item 1}
   - {item 2}

   Pipeline status: READY / NOT READY
   ```

8. If NOT READY, present specific blockers and stop.

### Success: All phases complete, all qa reports PASS or CONDITIONAL PASS, no unresolved blockers
### Failure: Any unchecked phase tasks, any qa FAIL verdict, phases.md missing

→ Present result. HALT. Wait for user before next stage.

### Stage 2: Release Checklist

1. Evidence collection — each check produces tangible proof (command output, search results, audit reports), not claimed status. Run comprehensive pre-deployment checks:

   **Build & Test:**
   If `phases.md` → `## Project Strategy` contains `Stack Pack:`, read the referenced
   stack pack from `~/.claude/skills/_shared/references/stacks/` and use its Toolchain
   section for build/test/lint/audit commands. Otherwise use these defaults:
   - Full build: `npm run build` / `pnpm build` / project-appropriate command
   - Full test suite: `npm test` / `pnpm test`
   - Typecheck: `npx tsc --noEmit` (if TypeScript project)
   For non-JS projects without a stack pack: check manifest for build commands
   (Cargo.toml → `cargo build`, go.mod → `go build ./...`, pyproject.toml → project scripts).

   **Code Quality:**
   - Search for leftover debug artifacts:
     - `console.log` / `console.debug` / `console.warn` (flag non-intentional)
     - `debugger` statements
     - `TODO` / `FIXME` / `HACK` / `XXX` comments
     - Hardcoded `localhost`, `127.0.0.1`, development URLs
   - Check for development flags: `NODE_ENV !== 'production'` guards, feature flags set to dev values
   - AI traces in tracked files (BLOCKING if repo may go public):
     - `.claude/` directory must NOT be in local `.gitignore` (belongs in global gitignore only — listing it in the repo's `.gitignore` reveals AI tool usage)
     - Search committed/tracked files for: "Generated with Claude", "Co-Authored-By: Claude", "AI-generated", "written by Claude", "Claude Code", "AI Studio" (outside discovery docs)
     - Check git log for Co-Authored-By lines referencing AI

   **Security:**
   - Run dependency vulnerability scan: use stack pack's Toolchain audit command if loaded
     (e.g., `cargo audit` for Rust, `pip-audit` for Python, `govulncheck ./...` for Go),
     otherwise fall back to `npm audit` / `pnpm audit`
   - Verify `.env.example` documents all required env vars from key-learnings
   - Check `.gitignore` includes `.env`, credentials, secrets

   **Legal:**
   - Verify `LICENSE` file exists at project root
   - If `package.json` has a `"license"` field, verify it matches the LICENSE file type (e.g., "MIT" → file should contain "MIT License")
   - Verify LICENSE file contains a copyright holder name (not blank or placeholder like "Copyright (c) 2026" with no name)
   - For npm publish: BLOCKING if LICENSE is missing (npm requires it for public packages)
   - For public repos: WARNING if LICENSE is missing (code defaults to "all rights reserved")
   - For private/internal: informational only

   **Environment:**
   - Compare `.env.example` against env vars referenced in code (Grep for `process.env.`)
   - Flag any env var in code that's not in `.env.example`
   - For Vercel: check `vercel.json` for region, runtime, and function configuration
   - For npm: verify `package.json` version, `files` field, `main`/`exports` fields

2. Generate `release-checklist.md` at project root using `assets/release-checklist-template.md` as the base template:

   ```markdown
   # Release Checklist — {version or date}

   ## Build & Test
   - [x/!] Build: {PASS/FAIL with output}
   - [x/!] Tests: {PASS/FAIL — N/N passed}
   - [x/!] Typecheck: {PASS/FAIL}

   ## Code Quality
   - [x/!] No debug artifacts: {PASS / N items found}
   - [x/!] No hardcoded dev values: {PASS / N items found}

   ## Security
   - [x/!] Dependency audit: {PASS / N vulnerabilities}
   - [x/!] Env vars documented: {PASS / N missing}
   - [x/!] Secrets excluded from git: {PASS/FAIL}

   ## Environment
   - [x/!] Env var coverage: {PASS / N undocumented}
   - [x/!] Platform config valid: {PASS/FAIL}

   ## Outstanding Items (from qa)
   - {items from CONDITIONAL PASS reports}

   ## Hotfixes Included
   - {hotfix entries since last deploy}
   ```

3. Present the checklist to the user. Apply these decision rules to each finding:

   **Severity classification**: When a finding is ambiguous (neither clearly blocking nor clearly cosmetic), apply: "Could a real user hit this in the first 5 minutes?" YES → mark as blocking. NO → document as caveat.

   **Fix vs ship**: Default to shipping with documented caveats UNLESS the finding touches data integrity or security → those must be fixed before deploying. A delayed deploy costs more than a documented caveat for most issues.

   If any blocking items remain (build failure, test failure, data integrity/security findings), recommend fixing before proceeding.

### Success: Build passes, tests pass, no critical security findings, checklist generated
### Failure: Build fails, tests fail, critical vulnerabilities found with no plan to address

→ Present result. HALT. Wait for user before next stage.

### Stage 3: Changelog

1. Read all key-learnings files chronologically. For each phase, extract:
   - Features added (from Summary and Architecture Decisions)
   - Changes to existing behavior (from Patterns Established)
   - Bugs fixed (from Issues Encountered)
   - Dependencies added/changed (from Dependencies & Versions Locked)

2. Read `fix-log.md` for fixes applied outside the pipeline (also check `hotfix-log.md` for legacy data).

   **IMPORTANT**: If `fix-log.md` exists and has entries, OR if any qa report
   documents fixes that were applied, the changelog MUST include a `### Fixed`
   section. Convert each fix-log entry into a user-facing description of what
   was fixed. Group by category where logical (accessibility, i18n, performance,
   security, etc.). Do NOT omit the Fixed section even if no new features were added.

3. Read qa reports for issues found and fixed (from Post-qa Fixes sections).

4. Detect existing changelog format:
   - If `CHANGELOG.md` exists, match its format
   - If not, use Keep a Changelog format

5. Generate the changelog entry:

   ```markdown
   ## [{version}] - {YYYY-MM-DD}

   ### Added
   - {new features, described for end users}

   ### Changed
   - {modifications to existing features}

   ### Fixed
   - {bug fixes from qa and fixes — REQUIRED if fix-log.md has entries}

   ### Security
   - {security-related changes}

   ### Dependencies
   - {notable dependency changes}
   ```

6. Write or update `CHANGELOG.md` at project root.

7. Present the changelog entry to the user for review. Incorporate edits if requested.

### Success: Changelog entry generated and written, user has reviewed
### Failure: Unable to read key-learnings or determine version

→ Present result. HALT. Wait for user before next stage.

### Stage 4: Deployment

**Blocking Conditions** — stop immediately if any are true:
- Build failed in Stage 2
- User has not explicitly confirmed the deployment command
- No deployment target detected and user has not specified one

1. Detect deployment target from project configuration:
   - `vercel.json` → Vercel deployment
   - `Dockerfile` → Docker/container deployment
   - `package.json` with `"publishConfig"`, `"bin"`, or `"main"`/`"exports"` (and not `"private": true`) → npm publish
   - `netlify.toml` → Netlify deployment
   - `fly.toml` → Fly.io deployment
   - None detected → ask user

2. **Git clean check**: Run `git status`. If there are uncommitted changes to files that affect the published output (package.json, source files, CHANGELOG.md), they must be committed before deploying. Present the list and ask the user to commit or confirm.

3. **npm-specific pre-publish checks** (skip for non-npm targets):
   a. **Name availability**: Run `npm publish --dry-run`. This catches both taken names AND npm's similarity check (e.g., `reviewkit` rejected for being too similar to `review-kit`). `npm view` returning 404 does NOT guarantee the name is publishable.
   b. **Auth verification**: Run `npm whoami`. If not logged in, prompt user to run `npm login`. See `deployment-targets.md` for auth details including granular token setup.
   c. If name or auth fails, resolve before proceeding.

4. Present the deployment command and confirm:
   ```
   Deployment Target: {target}
   Command: {exact command to run}
   Environment: {production / staging / preview}

   This will deploy to {target}. Proceed?
   ```

5. Launch commit gate — this is the point of no return. Wait for explicit user confirmation.

6. Execute the deployment command:
   - For Vercel: `vercel --prod` (or `vercel` for preview)
   - For npm: `npm publish` (see `deployment-targets.md` for auth token handling)
   - For Docker: `docker build && docker push`
   - Capture full output

5. Report deployment result:
   ```
   Deployment: {SUCCESS / FAILED}
   {deployment URL if applicable}
   {error output if failed}
   ```

6. If deployment fails: present the error and stop. Do not auto-retry deployments.

### Success: Deployment command exits successfully, URL or confirmation received
### Failure: Deployment command fails, error output present

→ Present result. HALT. Wait for user before next stage.

### Stage 5: Post-Deploy Verification

1. If a deployment URL is available (Vercel, Netlify, etc.):

   **Automated checks (via Playwright MCP if available):**
   - Navigate to the deployed URL
   - Verify the page loads without errors
   - Check `browser_console_messages` for errors
   - Check `browser_network_requests` for failed requests
   - Navigate to key routes identified from phases.md
   - Take screenshots as evidence

   **Smoke test subset:**
   - From qa Category C (user journeys), run the 2-3 most critical journeys
   - Verify core functionality works in the deployed environment
   - Check for CORS issues (common in Vercel serverless deployments)

2. If no URL is available (npm package, CLI tool, library):
   - Verify the package was published: `npm view {package}@{version}`
   - Create a temp directory, `npm init -y`, `npm install {package}@{version}`
   - If CLI (`bin` field): run `npx {bin} --version`, `npx {bin} --help`, and test error handling (e.g., missing file → exit 2)
   - If library: `node -e "import('{package}')"` (ESM) or `node -e "require('{package}')"` (CJS)
   - If CLI with API key dependency: test that missing-key path produces a clear error, and if key is available, run one real command
   - Clean up temp directory after verification

3. Present post-deploy verification report:
   ```
   Post-Deploy Verification: {URL or package}

   | Check | Status | Evidence |
   |-------|--------|----------|
   | Page loads | PASS | {screenshot or response code} |
   | No console errors | PASS | {console output} |
   | No failed requests | PASS | {network log} |
   | Core journey 1 | PASS | {description} |
   | Core journey 2 | PASS | {description} |

   Deployment verified: {YES / NO — issues found}
   ```

4. If verification fails, classify and recommend:
   - **User-visible breakage** (page won't load, core journey broken, data corruption) → recommend rollback
   - **Backend-only issues** (elevated error rates, slow responses, non-critical API failures) → recommend monitoring, not rollback
   Present options to the user. Do not auto-rollback.

5. **Knowledge capture** (lightweight):
   - Read `~/.claude/skills/_shared/deep-knowledge.md` (if exists)
   - Check: did anything in this deployment contradict or extend a deep-knowledge entry? If so, note in the final summary for user review.

6. **Pipeline archival**: Archive `pipeline-state.md` (if it exists): ask the user via `AskUserQuestion` whether to archive (rename to `pipeline-state-{YYYY-MM-DD}.md` in the project root) or delete. This clears the state for the next pipeline cycle — the next plan invocation will create a fresh `pipeline-state.md`. If `pipeline-state.md` does not exist, skip this step.

7. **Port release**: Read `~/.claude/skills/_shared/references/port-registry.md` (if exists). Find this project's entry and update its Status from `active` to `released`. This frees the port for reuse by future projects. If the file does not exist or the project has no entry, skip silently.

8. Present final deployment summary:
   ```
   deploy Complete

   Target: {deployment target}
   URL: {URL if applicable}
   Version: {version if applicable}
   Release checklist: release-checklist.md
   Changelog: CHANGELOG.md
   Verification: {PASS / ISSUES FOUND}

   Artifacts created:
   - release-checklist.md
   - CHANGELOG.md (updated)
   ```

Next steps: `/fix` for any post-deploy findings or urgent production issues.

### Success: Deployed URL verified, no critical console errors, core journeys pass
### Failure: Page fails to load, critical errors in console, core journeys broken

→ Present result. HALT. Wait for user before next stage.

## When I Hit My Limits

1. Check `brain.md` Gaps — if this is a known limit, work around it and note it
2. Unfamiliar deployment target not covered by `deployment-targets.md` → emit request to `~/.claude/skills/_shared/requests/pending/` with format: `{date}-deploy-{target-type}.md` describing the platform, what commands were attempted, and what documentation is needed
3. Platform error not matching known patterns in any reference → invoke deep-researcher: `"deployment error {error} for {platform} — diagnosis, root cause, and resolution steps"`
4. Mark affected checks as `[UNVERIFIED]` and continue — a partial checklist is better than no checklist
